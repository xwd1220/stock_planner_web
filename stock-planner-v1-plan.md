# 股票交易计划工具 V1 实现方案

## 1. 一期范围

V1 仅包含以下能力：

- 标的配置
- 买入计划生成器
- 卖出计划生成器

V1 明确不包含以下能力：

- 行情抓取
- VIX 监控
- 回撤监控
- 提醒与通知
- 自动交易
- DCF 估值自动化

## 2. 产品目标

做一个本地或内部使用的 Web 工具，帮助用户完成以下事情：

1. 配置股票或 ETF 的基础信息与交易参数
2. 根据初始价格和分批规则生成买入计划
3. 根据持仓成本、目标回收资金和卖出规则生成卖出计划
4. 清晰查看总投入、总股数、持仓成本、回收资金和剩余仓位

## 3. 核心业务规则

### 3.1 标的配置

每个计划对应一个标的，基础字段建议包括：

- 标的代码，如 `NVDA`、`QQQ`
- 标的名称
- 标的类型：`个股`、`ETF`
- 币种，默认 `USD`
- 备注

### 3.2 买入计划规则

输入项：

- 初始买入价格
- 计划资金
- 买入频率百分比
- 买入次数
- 首单数量
- 买单倍数列表

计算规则：

- 第 1 节点数量 = 首单数量
- 下一节点买入价格 = 上一节点价格 × `（1 - 买入频率）`
- 下一节点买入数量 = 上一节点数量 × 当前节点倍数
- 单节点消耗资金 = 节点价格 × 节点数量
- 累计投入资金 = 各节点消耗资金之和
- 累计买入数量 = 各节点数量之和
- 持仓成本 = 累计投入资金 ÷ 累计买入数量

校验规则：

- 买入次数必须等于买单倍数数量
- 价格、数量、倍数必须大于 0
- 需要比较“计划资金”和“实际累计投入资金”
- 如果累计投入超过计划资金，需要标记为“超预算”

### 3.3 卖出计划规则

输入项：

- 持仓成本
- 总持仓数量
- 计划卖出数量
- 目标回收资金
- 初始卖出价格
- 初始执行价格
- 卖出频率百分比
- 卖出次数
- 卖单倍数列表

规则说明：

- 卖出价格用于计算“回收资金”
- 执行价格用于计算“计划资金”
- 下一节点卖出价格 = 上一节点卖出价格 × `（1 + 卖出频率）`
- 下一节点执行价格 = 上一节点执行价格 × `（1 + 卖出频率）`
- 卖出数量按倍数权重分配，不按上一节点数量连乘
- 每节点权重 = 当前倍数 ÷ 全部倍数之和
- 每节点原始卖出数量 = 计划卖出数量 × 当前权重
- 因为股数通常为整数，前面节点按规则取整，最后一个节点吸收舍入差值
- 节点回收资金 = 卖出价格 × 节点卖出数量
- 节点计划资金 = 执行价格 × 节点卖出数量
- 剩余持仓 = 总持仓数量 - 累计卖出数量

校验规则：

- 卖出次数必须等于卖单倍数数量
- 计划卖出数量必须大于 0，且不能超过总持仓数量
- 价格和倍数必须大于 0

## 4. 产品形态建议

当前工作区已有项目是招投标爬虫，与本次需求无直接关系，因此建议将 V1 作为独立应用开发，或者至少作为清晰隔离的子模块开发。

推荐技术方案：

- 前端：React
- 后端：FastAPI
- 数据库：SQLite

如果目标是尽快验证公式和交互，也可以先用：

- Streamlit + Python 服务层 + SQLite

建议选择：

- 如果优先验证业务逻辑，选 `Streamlit`
- 如果希望后续平滑扩展到监控、提醒、历史记录，选 `React + FastAPI`

## 5. 推荐系统架构

### 5.1 前端

页面建议：

- 标的列表页
- 新建或编辑标的计划页
- 买入计划结果页
- 卖出计划结果页
- 计划详情页

前端职责：

- 收集用户输入
- 做基础表单校验
- 展示买入和卖出结果表格
- 高亮超预算、数量不匹配等提示

### 5.2 后端

模块建议：

- `symbols`：标的配置
- `buy_plans`：买入计划计算
- `sell_plans`：卖出计划计算
- `shared`：通用校验、四舍五入、数据结构定义

后端职责：

- 保存计划配置
- 执行买入和卖出计算逻辑
- 返回标准化结果
- 保证计算逻辑与界面解耦

### 5.3 存储

V1 使用 SQLite 足够，原因如下：

- 适合单用户或小团队
- 数据结构简单
- 易于备份和迁移

## 8. 计算服务设计

建议把所有公式都写成独立的纯计算服务，避免和页面逻辑耦合。这样做有几个好处：

- 后续换前端时不影响核心规则
- 单元测试更容易覆盖
- 未来要增加行情监控时可以直接复用

建议模块：

- `services/buy_plan.py`
- `services/sell_plan.py`
- `services/rounding.py`

## 10. 校验与边界情况

需要重点处理以下情况：

- 倍数数量与次数不一致
- 输入为 0 或负数
- 买入计划累计投入超过计划资金
- 卖出计划的计划卖出数量超过总持仓
- 标的代码为空
- 小数股是否允许

建议一期默认规则：

- 价格保留 2 位小数
- 金额保留 2 位小数
- 股数默认按整数处理

如果后续要支持碎股，可以把“股数精度”做成可配置项。

## 11. 测试方案

### 11.1 单元测试

重点覆盖：

- 买入价格阶梯计算
- 买入数量按“上一节点数量 × 当前倍数”计算
- 持仓成本计算
- 卖出数量按权重分配
- 卖出最后一个节点吸收舍入误差
- 各类输入校验失败场景

### 11.2 集成测试

建议覆盖：

- 新建标的 -> 计算买入计划 -> 保存买入计划
- 新建标的 -> 计算卖出计划 -> 保存卖出计划

## 13. 建议的开发顺序

如果现在就要开工，最稳妥的顺序是：

1. 先实现后端计算服务
2. 用需求中的示例数据固化测试用例
3. 再补最小可用页面
4. 最后接入持久化存储

这样可以把最核心、最容易返工的“计算口径”先锁定。
